# Pipeline Bug Fixes — 31 May 2026

Bugs found and fixed during the rework of `scripts/wearables/` and `scripts/analysis/`
into `scripts/extraction/`. Documented for reference.

---

## Fix 1 — Hardcoded UTC offset ignores CEST

**File:** `scripts/wearables/garmin_pipeline.py` and `huawei_pipeline.py`
**Function:** `crossref_sessions()`

### Problem

The local-to-UTC conversion used a fixed integer offset:

```python
sessions[col] = sessions.apply(
    lambda r: pd.Timestamp(f"{r['_date'].date()} {r[src]}") - pd.Timedelta(hours=utc_offset),
    axis=1,
)
# utc_offset defaults to 1 — always subtracts 1 hour
```

Belgium observes CET (UTC+1) in winter and CEST (UTC+2) in summer.
The DST transition for 2026 was **Sunday 29 March at 02:00**.
Every session after that date had its biometric window shifted 1 hour forward in UTC,
meaning the pipeline was reading biometrics from the wrong time slot.

### Impact (confirmed on peer)

| Session | Old stress_points | New stress_points |
|---|---|---|
| 2026-04-15 | 9 | 31 |
| 2026-04-20 | 10 | 29 |
| 2026-05-08 | **0** | 31 — old pipeline found nothing |
| 2026-05-10 | 1 | 32 |
| 2026-05-14 | 13 | 31 |

Pre/during/post stress values also shifted substantially for all CEST sessions.

### Fix

Replaced the fixed offset with `zoneinfo.ZoneInfo("Europe/Brussels")`, which
handles CET/CEST transitions automatically:

```python
from zoneinfo import ZoneInfo
BE = ZoneInfo("Europe/Brussels")

def local_to_utc(date, time_str: str) -> pd.Timestamp:
    local_dt = pd.Timestamp(f"{date} {time_str}").tz_localize(BE)
    return local_dt.tz_convert("UTC").tz_localize(None)
```

Also fixed the reverse conversion (UTC → local display) for `start_local`/`end_local`
columns in session_biometrics.csv.

**Now in:** `scripts/extraction/utils.py` — `local_to_utc()` and `utc_to_local()`

## Fix 2 — FIT activity extraction silently returns empty DataFrame

**File:** `scripts/analysis/fit_extractor.py`
**Function:** `extract_monitoring_activity()`

### Problem

The `activity_type` field in Garmin FIT monitoring messages is decoded by
fitparse as a string (e.g. `'generic'`, `'walking'`, `'sedentary'`) rather
than a raw integer. The code assumed an integer and called `int()` on it:

```python
raw_at = f.get("activity_type")
if raw_at is not None:
    activity_type = _ACTIVITY_MAP.get(int(raw_at), str(raw_at))  # raises ValueError
```

`int('generic')` raises `ValueError`. This was caught by a bare
`except Exception: pass` wrapping the entire per-file loop, silently
abandoning every FIT file. Result: 0 rows, empty DataFrame,
no `garmin_minute_activity.csv` written.

The same bug affected the `intensity` field.

### Impact

`garmin_minute_activity.csv` was never produced for participants whose
FIT monitoring messages used pre-decoded string values (confirmed on peer:
11,331 FIT files all affected). Activity classification subsequently ran
without the `intensity` and `activity_type` signal columns.

### Fix

Check the type before converting:

```python
if raw_at is not None:
    activity_type = (
        _ACTIVITY_MAP.get(raw_at, str(raw_at))
        if isinstance(raw_at, int)
        else str(raw_at)  # fitparse already decoded it
    )
```

**Now in:** `scripts/extraction/fit_extractor.py`

## Verification (Pipeline 1)

Both fixes were validated on participant **peer** (Garmin, 30 sessions,

- `garmin_minute_activity.csv`: 948 monitoring records extracted (was 0)
- `classified_minutes.csv`: 240,858 records, 236,186 classified
- Pre-CEST sessions: values identical to old pipeline (no regression)

## Fix 3 — Same hardcoded UTC offset in feature matrix pre-state lookup

**File:** `scripts/analysis/circadian_baseline.py` → now `scripts/baseline/circadian_baseline.py`
**Function:** `build_feature_matrix()`

### Problem

The same fixed `-1 hour` offset from Fix 1 was also present in `build_feature_matrix()`,
used when looking up the pre-session activity state from `classified_minutes.csv`:

```python
start_local = pd.Timestamp(f"{row['date'].date()} {row['start_local']}")
start_utc = start_local - pd.Timedelta(hours=1)  # CET → UTC  ← always -1 hour
pre_state = _classify_window_state(
    classified_df,
    start_utc - pd.Timedelta(minutes=30),
    start_utc,
)
```

For any session after the DST transition (29 March), `start_utc` was 1 hour too late,
meaning the 30-minute pre-session activity window was read from the wrong time slot.
This affects the `pre_state` and `pre_state_encoded` columns in `feature_matrix.csv`.

### Fix

Replaced with `local_to_utc()` from `baseline/utils.py` (same zoneinfo fix as Fix 1):

```python
start_utc = local_to_utc(row["date"].date(), row["start_local"])
pre_state = _classify_window_state(
    classified_df,
    start_utc - pd.Timedelta(minutes=30),
    start_utc,
)
```

**Now in:** `scripts/baseline/utils.py` — `local_to_utc()` (same implementation as `scripts/extraction/utils.py`)
